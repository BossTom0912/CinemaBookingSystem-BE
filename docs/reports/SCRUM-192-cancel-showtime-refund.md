# SCRUM-192 - Manager Cancel Showtime and Refund Processing

## 1. Task Scope

This implementation focuses on the Manager workflow:

- cancel a showtime in the Manager's assigned cinema;
- stop all new seat selection and ticket sales;
- cancel unpaid bookings;
- move paid bookings to refund processing;
- create traceable refund records;
- attempt automatic refund processing through an abstraction;
- move unsupported or failed automatic refunds to manual processing;
- expose the Manager refund list within the assigned cinema;
- notify customers and keep audit history.

Admin-specific workflow and tests are not expanded in this task. The existing shared
authorization policy still permits Admin because the authoritative SRS grants Admin
system-wide cancellation permission.

## 2. Authoritative Sources

### 2.1 SRS

Source: `docs/requirements/srs-group-2.docx`

UC003, `Cancel Showtime and Auto-Refund`, requires:

1. Manager selects a showtime and provides a reason.
2. The backend verifies cinema permission.
3. The showtime becomes `CANCELLED`.
4. New ticket sales are blocked.
5. Paid bookings are found and moved into refund processing.
6. Booking and ticket states are updated.
7. Eligible payments are sent to the payment/refund phase.
8. Customers are notified.
9. Cancellation and refund results are audited.

The UC003 alternative and exception flows state:

- an unpaid booking is cancelled without a refund;
- an unsupported refund provider requires manual processing;
- a gateway failure requires manual processing;
- notification failure must be logged;
- a Manager cannot cancel a showtime outside the assigned cinema.

### 2.2 Business Rules

Source: `docs/requirements/business-rules.docx`

- BR-38: paid bookings of a cancelled showtime become `REFUND_PENDING`.
- BR-39: a successful refund makes the booking `REFUNDED`.
- BR-40: an unsuccessful automatic refund requires manual handling.
- BR-41: the customer must be notified when a booking is cancelled or refunded.
- BR-60: reward points generated by a refunded booking must be reverted or invalidated.
- BR-75: resources with paid booking history cannot be deleted outside the cancellation
  workflow.
- BR-79: Manager can access only data from the assigned cinema.
- BR-82: cancellation and refund operations must be audited.

### 2.3 Backend Design

Source: `docs/architecture/backend-system-design-clean-architecture.docx`

Section 10.6 defines two boundaries:

```text
Phase 1 - database cancellation transaction
Phase 2 - payment/refund processing
```

The document explicitly requires:

- a transaction for showtime, booking, ticket, refund, and audit changes;
- payment gateway work after refund data exists;
- `SUCCESS` to update refund/booking;
- failure to move the refund to manual processing;
- a separate application abstraction from concrete gateway code;
- audit entries for both cancellation and refund processing.

## 3. Architecture

### API layer

- `ShowtimeCancellationsController`
  - receives the cancellation request;
  - enforces the authorization policy;
  - resolves Manager cinema scope;
  - delegates to `IShowtimeCancellationService`.
- `ManagerRefundsController`
  - resolves Manager cinema scope;
  - delegates refund queries to `IRefundService`.

Controllers do not perform EF Core queries or refund business logic.

### Application layer

- `IShowtimeCancellationService`
- `IRefundService`
- `IRefundProcessor`
- `IPaymentRefundGateway`
- gateway request/result models

The gateway abstraction prevents the cancellation use case from depending directly on
SePay or another provider.

### Infrastructure layer

- `ShowtimeCancellationService`
  - owns the cancellation transaction;
  - creates refund data;
  - triggers post-commit processing.
- `RefundProcessor`
  - owns refund state transitions;
  - updates booking/ticket/reward/notification/audit data.
- `UnsupportedPaymentRefundGateway`
  - reports that automatic refund is unavailable.
- `RefundService`
  - returns scoped Manager refund data.

## 4. Stage 1 - Manager Authorization and Validation

Endpoint:

```http
POST /api/manager/showtimes/{showtimeId}/cancel
```

Request:

```json
{
  "reason": "Projector failure"
}
```

Validation:

- `showtimeId` is required;
- `reason` is required and at most 1000 characters;
- showtime must exist;
- showtime must not already be cancelled;
- showtime must not be completed;
- the authenticated Manager must have an active `STAFF_PROFILE`;
- `STAFF_PROFILE.cinemaId` must match the show's room cinema.

Relevant errors:

```text
SHOWTIME_NOT_FOUND
SHOWTIME_ALREADY_CANCELLED
SHOWTIME_NOT_CANCELLABLE
CANCEL_REASON_REQUIRED
CANCEL_REASON_TOO_LONG
CINEMA_SCOPE_FORBIDDEN
STAFF_PROFILE_SCOPE_NOT_FOUND
```

Why scope comes from the token/profile:

- BR-79 prevents Manager access outside the assigned cinema.
- The request cannot submit a different `cinemaId`.
- Scope is resolved from the authenticated user and active staff profile.

## 5. Stage 2 - Cancellation Database Transaction

`ShowtimeCancellationService` opens one database transaction.

### 5.1 Showtime state

```text
SHOWTIME.status = CANCELLED
```

One `SHOWTIME_CANCELLATION` is created with:

- showtime ID;
- cancelling user;
- active staff profile when available;
- cancellation reason;
- UTC cancellation time.

The unique showtime cancellation constraint provides the final duplicate guard.
SQL Server duplicate-key errors are mapped to:

```text
409 SHOWTIME_ALREADY_CANCELLED
```

### 5.2 Seat state

Every showtime seat becomes:

```text
seatStatus = UNAVAILABLE
lockedUntil = NULL
lockedByUserId = NULL
```

`SeatService` also rejects seat map and lock operations when the showtime is not `OPEN`.
This provides defence in depth: both showtime state and showtime-seat state block sales.

### 5.3 Unpaid booking state

For `CREATED` or `PENDING_PAYMENT` bookings:

```text
BOOKING = CANCELLED
PAYMENT.PENDING = CANCELLED
TICKET, if present = CANCELLED
```

No refund is created because no successful payment exists.

### 5.4 Paid booking state

For `PAID` bookings:

```text
BOOKING = REFUND_PENDING
TICKET = CANCELLED
```

The latest successful payment is selected and one refund is created:

```text
REFUND.status = PENDING
REFUND.amount = successful PAYMENT.amount
REFUND.showtimeCancellationId = cancellation ID
```

An existing active refund prevents duplicate data. Active means:

```text
PENDING
SUCCESS
MANUAL_REQUIRED
```

### 5.5 Cancellation notification and audit

The transaction creates:

- internal customer notifications;
- `AUDIT_LOG.action = CANCEL_SHOWTIME`.

The audit payload includes:

- old/new showtime status;
- cancellation ID;
- reason;
- paid booking count;
- unpaid booking count;
- refund count and amount.

### 5.6 Commit boundary

The database transaction commits before:

- SMTP email;
- payment gateway calls.

Reason:

- a slow gateway must not hold SQL locks;
- email/gateway failure must not reopen the cancelled showtime;
- ticket sales must remain blocked once cancellation succeeds.

Code comments are placed at this boundary in
`ShowtimeCancellationService`.

## 6. Stage 3 - Customer Cancellation Email

After the cancellation commit:

- online booking email comes from `CustomerProfile.User.Email`;
- counter booking email falls back to `Booking.GuestEmail`.

SMTP failure is caught and logged. It does not roll back cancellation.

Internal notification remains stored even when email cannot be sent.

The current project has no email outbox/retry policy for this flow. No retry interval or
maximum attempt count is invented because the SRS does not define them.

## 7. Stage 4 - Refund Processing

Each newly created refund is passed to `IRefundProcessor` after cancellation commits.

### 7.1 Gateway boundary

`IPaymentRefundGateway` receives:

- refund ID;
- original payment ID;
- provider ID/name;
- refund amount;
- original provider transaction code;
- refund reason.

The refund ID is the provider idempotency key expected from a future concrete adapter.

The external call is outside the EF transaction.

### 7.2 Current SePay capability

Current `SepaySettings` contains:

- webhook secret;
- bank name;
- bank account;
- development payment override.

It contains no:

- refund endpoint;
- refund credential;
- refund signature configuration;
- refund callback contract.

Therefore `UnsupportedPaymentRefundGateway` returns `Unsupported`.

The application does **not** claim that money was returned. The refund becomes:

```text
REFUND = MANUAL_REQUIRED
BOOKING = REFUND_PENDING
TICKET = CANCELLED
```

This implements the UC003 alternative flow and BR-40 honestly.

### 7.3 Future supported gateway success

The processor already supports a real adapter returning success:

```text
REFUND = SUCCESS
providerRefundCode = provider result
refundedAt = UTC now
```

When total successful refunds cover the original successful payment:

```text
BOOKING = REFUNDED
TICKET = REFUNDED
```

The processor prevents successful refund totals from exceeding the original payment.

### 7.4 Idempotency

Before calling the gateway:

- `SUCCESS` returns as already processed;
- `MANUAL_REQUIRED` returns as already processed;
- only `PENDING` can be processed.

A repeated payment webhook also attempts to resume an existing pending late-payment
refund instead of returning before refund processing.

### 7.5 Late successful payment

If payment succeeds after cancellation:

```text
PAYMENT = SUCCESS
BOOKING = REFUND_PENDING
SHOWTIME_SEAT = UNAVAILABLE
TICKET = CANCELLED
REFUND = PENDING
```

The refund is then passed through the same processor. With current provider
configuration it becomes `MANUAL_REQUIRED`.

This prevents a late callback from reopening the booking or generating a usable ticket.

## 8. Stage 5 - Refund Completion Side Effects

### Successful refund

The processor creates:

- customer notification: refund completed;
- customer email after commit;
- `PROCESS_REFUND` audit entry;
- reward-point reversal when applicable.

### Manual-required refund

The processor creates:

- customer notification explaining additional processing;
- customer email after commit;
- `PROCESS_REFUND` audit with failure reason.

The booking remains `REFUND_PENDING` because the schema's booking status enum does not
contain `MANUAL_REQUIRED`; that state belongs to `REFUND`.

## 9. Stage 6 - Reward Point Reversal

Source: BR-60.

When a booking becomes fully refunded:

1. Sum positive `EARN` transactions linked to the booking.
2. Subtract reward points already represented by negative `REVERT` transactions.
3. Create one negative `REVERT` transaction for the outstanding earned amount.
4. Reduce the profile balance without allowing it below zero.

The ledger records full invalidation even if part of the earned balance was spent before
refund. Reprocessing the same refund does not create another reversal.

Voucher usage is not changed because the supplied business rules define voucher
behaviour for payment timeout/failure but do not define voucher restoration after
refund.

## 10. Stage 7 - Manager Refund List

Endpoint:

```http
GET /api/manager/refunds
```

Optional query parameters:

```text
status
showtimeId
from
to
```

Manager scope is applied before other filters:

```text
REFUND
-> BOOKING
-> SHOWTIME
-> ROOM.cinemaId
```

Response includes:

- refund, booking, payment, and showtime IDs;
- payment provider ID/name;
- cinema ID/name;
- movie title;
- refund and booking status;
- amount;
- reason and failure reason;
- provider refund code;
- requested/refunded time.

This gives Manager enough data to identify `MANUAL_REQUIRED` records without exposing
another cinema.

## 11. Stage 8 - Audit History Protection

A showtime with `SHOWTIME_CANCELLATION` can no longer be permanently deleted, even if it
has no booking or refund.

Error:

```text
409 RESOURCE_HAS_CANCELLATION_HISTORY
```

Reason:

- the cancellation entity records actor, reason, and time;
- deleting it would violate the audit purpose described by the ERD and BR-82.

## 15. Replacing the Placeholder Gateway

When a payment provider supplies an official refund API:

1. Create a concrete `IPaymentRefundGateway` adapter.
2. Use `RefundId` as the external idempotency key.
3. Sign requests using provider requirements.
4. Return `PaymentRefundGatewayResult.Success(providerRefundCode)` only after confirmed
   provider acceptance/success.
5. Register the concrete adapter instead of `UnsupportedPaymentRefundGateway`.
6. Add provider contract and callback tests.

No change to cancellation controller or cancellation transaction should be required.
