TẬP 1: KIẾN TRÚC HỆ THỐNG (CLEAN ARCHITECTURE SIMPLIFIED)
================================================================================
Dự án được xây dựng theo mô hình Clean Architecture giản lược (Simplified Clean Architecture) kết hợp Layered Architecture. Hệ thống được chia thành 6 dự án thành phần (nhãn hiệu suất cao, dễ bảo trì và mở rộng):

1. CinemaSystem (Tầng API - Trực quan/Giao tiếp bên ngoài)
   - Vai trò: Tiếp nhận HTTP Request từ Client (Web/Mobile App), kiểm tra tính hợp lệ cơ bản của dữ liệu (Model Validation) và trả về HTTP Response (JSON).
   - Thành phần chính: Controllers (AuthController, CinemasController, v.v.), Middlewares (GlobalExceptionMiddleware), Swagger Configuration.
   - Nguyên tắc: Không chứa Logic nghiệp vụ nặng hay truy vấn SQL trực tiếp. Chỉ điều phối luồng bằng cách gọi các Service của tầng Application.

2. CinemaSystem.Application (Tầng Nghiệp vụ - Đầu não hệ thống)
   - Vai trò: Định nghĩa luật nghiệp vụ (Business Rules), các hằng số (Constants) và các hợp đồng giao tiếp (Interfaces).
   - Thành phần chính: IAuthService, IJwtTokenService, IEmailSender, IPasswordHasher.
   - Nguyên tắc: Tầng này chỉ biết "hệ thống cần làm những gì" (ví dụ: cần mã hóa mật khẩu) chứ không biết "làm bằng cách nào" (không biết dùng thư viện nào). Điều này giúp hệ thống độc lập hoàn toàn với công nghệ bên ngoài.

3. CinemaSystem.Infrastructure (Tầng Hạ tầng - Công cụ thực thi)
   - Vai trò: Triển khai chi tiết các Interface từ Application bằng công nghệ cụ thể.
   - Thành phần chính: DbContext (EF Core giao tiếp với SQL Server), Pbkdf2PasswordHasher (Mã hóa mật khẩu), SmtpEmailSender (Gửi mail qua giao thức SMTP), JwtTokenService (Tạo mã JWT).
   - Nguyên tắc: Chứa toàn bộ các chi tiết kỹ thuật của hệ thống và các thư viện bên thứ ba.

4. CinemaSystem.Contracts (Tầng Hợp đồng DTO - Vận chuyển dữ liệu)
   - Vai trò: Định nghĩa các cấu trúc dữ liệu (Data Transfer Objects - DTO) để trao đổi giữa Client và API.
   - Thành phần chính: Request DTOs (LoginRequest, RegisterRequest) và Response DTOs (AuthResponse, TokenResponse).
   - Nguyên tắc: Tuyệt đối không trả trực tiếp các thực thể Database (Entity) ra Client nhằm tránh lộ thông tin nhạy cảm và giảm tải dữ liệu thừa.

5. CinemaSystem.Domain (Tầng Core/Mô hình thực thể - Lõi cơ sở)
   - Vai trò: Định nghĩa các thực thể (Entities) ánh xạ trực tiếp với bảng trong Database, cùng với các Enum và luật cốt lõi.
   - Thành phần chính: User, Role, CustomerProfile, Movie, Booking, v.v.

6. CinemaSystem.Tests (Tầng Kiểm thử - Đảm bảo chất lượng)
   - Vai trò: Chứa các kịch bản kiểm thử tự động (Unit Test & Integration Test) để kiểm tra độ chính xác của logic nghiệp vụ (AuthServiceTests, v.v.).

--------------------------------------------------------------------------------
* LUỒNG ĐI CỦA DỮ LIỆU (DATA FLOW):
  Client (Request) ──> API Controller ──> Application Interface (IAuthService)
                                                   │
                                                   ▼
  SQL Server Database <── EF Core DbContext <── Infrastructure Service (AuthService)
                                                   │
                                                   ▼
  Client (Response) <── API Controller <── Response DTO (AuthResponse)
--------------------------------------------------------------------------------

TẬP 2: THIẾT KẾ CƠ SỞ DỮ LIỆU & QUY TẮC NGHIỆP VỤ
================================================================================
Cơ sở dữ liệu của dự án gồm 30 bảng, chia thành 8 phân hệ nghiệp vụ chính (Xem chi tiết tại DATABASE_DEEP_DIVE_VI.md).

* Các quy tắc thiết kế hệ thống quan trọng:
1. Định danh UUID/Chuỗi (NVARCHAR(50)):
   - Khóa chính (PK) của toàn bộ các bảng trong hệ thống đều sử dụng kiểu NVARCHAR(50) thay vì kiểu INT tự tăng. Điều này giúp mã hóa ID dạng GUID hoặc chuỗi tự sinh có tiền tố (ví dụ: USR0001, BKG84920), tăng tính bảo mật, tránh việc kẻ xấu dò quét ID tuần tự (Id ID-harvesting attack).

2. Lưu trữ vết lịch sử giao dịch (Snapshot Pattern):
   - Giá vé (seatPrice), đơn giá sản phẩm (unitPrice) tại thời điểm khách hàng mua vé/F&B được chụp lại và lưu trực tiếp vào bảng BOOKING_SEAT hoặc BOOKING_FB_ITEM.
   - Lý do: Giá gốc trong danh mục (MOVIE, SEAT_TYPE, FB_ITEM) có thể thay đổi trong tương lai. Việc chụp lại giá giúp đảm bảo tính đúng đắn cho các báo cáo doanh thu và lịch sử thanh toán lịch sử không bị thay đổi.

3. Không sử dụng Cascade Delete (ON DELETE CASCADE) trên Database:
   - Toàn bộ các mối quan hệ khóa ngoại (FK) đều được thiết lập để chặn xóa (Restricted). Khi xóa một bản ghi cha (ví dụ: MOVIE), SQL Server sẽ chặn hành động đó nếu có dữ liệu con (ví dụ: SHOWTIME) đang tham chiếu đến nó. Điều này giúp ngăn ngừa mất mát dữ liệu tài chính vô ý. Thay vào đó, hệ thống sử dụng cờ trạng thái (Status/Active) để vô hiệu hóa phần tử.

TẬP 3: PHÂN TÍCH THUẬT TOÁN BẢO MẬT & ĐOẠN CODE QUAN TRỌNG
================================================================================

1. THUẬT TOÁN MÃ HÓA MẬT KHẨU (PBKDF2-SHA256)
   - File tham chiếu: Pbkdf2PasswordHasher.cs (Tầng Infrastructure)
   - Nguyên lý:
     Mật khẩu người dùng tuyệt đối không được lưu dưới dạng văn bản thuần túy (Plaintext). Hệ thống sử dụng thuật toán PBKDF2 (Password-Based Key Derivation Function 2) kết hợp hàm băm SHA-256 để tạo mã hash mật khẩu.
   
   - Các thuộc tính cấu hình thuật toán:
     + SaltSize = 16 bytes: Một chuỗi ngẫu nhiên sinh ra cho mỗi tài khoản để chống lại tấn công dùng bảng băm dựng sẵn (Rainbow Table). Dù hai người dùng đặt mật khẩu giống nhau, chuỗi băm lưu trong DB vẫn hoàn toàn khác nhau.
     + KeySize = 32 bytes: Độ dài khóa đầu ra (256-bit).
     + Iterations = 100,000 vòng: Số lần lặp để tính toán khóa băm. Con số 100k đủ lớn để làm chậm tốc độ thử mật khẩu hàng loạt của hacker (Brute-force attack), nhưng không gây trễ đáng kể cho người dùng thật (chỉ mất khoảng vài mili-giây).
     + Format lưu trữ: Lưu dưới dạng chuỗi ghép nối: "Algorithm.Iterations.Salt.Key" (Ví dụ: PBKDF2-SHA256.100000.bXNhbHQ=.a2V5aGFzaA==).

   - Đoạn Code quan trọng:
     ```csharp
     // Sinh salt ngẫu nhiên bảo mật
     var salt = RandomNumberGenerator.GetBytes(SaltSize);
     // Hàm băm PBKDF2
     var key = Rfc2898DeriveBytes.Pbkdf2(secret, salt, Iterations, HashAlgorithmName.SHA256, KeySize);
     ```

   - Kỹ thuật chống Timing Attack:
     Khi so sánh mật khẩu lúc đăng nhập, hệ thống sử dụng hàm `CryptographicOperations.FixedTimeEquals`.
     ```csharp
     return CryptographicOperations.FixedTimeEquals(actualKey, expectedKey);
     ```
     Ý nghĩa: So sánh chuỗi thông thường (như toán tử ==) sẽ dừng lại ngay khi phát hiện ký tự khác nhau đầu tiên, khiến thời gian phản hồi thay đổi nhỏ. Hacker có thể đo đạc thời gian phản hồi để đoán từng ký tự của mật khẩu. Hàm `FixedTimeEquals` luôn so sánh hết toàn bộ các byte bất kể đúng hay sai, giữ thời gian xử lý không đổi, triệt tiêu nguy cơ rò rì thông tin qua kênh thời gian (Timing Side-Channel Attack).

2. THUẬT TOÁN SINH MÃ OTP XÁC THỰC AN TOÀN
   - File tham chiếu: CryptoOtpGenerator.cs (Tầng Infrastructure)
   - Nguyên lý:
     OTP (One-Time Password) xác thực tài khoản hoặc khôi phục mật khẩu cần phải là số ngẫu nhiên không thể dự đoán được. Lớp `Random` thông thường của C# sinh số theo thuật toán giả ngẫu nhiên có chu kỳ (Pseudo-random), nếu biết hạt giống (seed) có thể đoán được chuỗi số tiếp theo.
     Hệ thống sử dụng lớp `RandomNumberGenerator` của thư viện mật mã học hệ điều hành để sinh số ngẫu nhiên thực sự.
   
   - Đoạn Code quan trọng:
     ```csharp
     public string GenerateSixDigitOtp()
     {
         // Sinh số ngẫu nhiên từ 000000 đến 999999 bảo mật tuyệt đối
         var value = RandomNumberGenerator.GetInt32(0, 1_000_000);
         return value.ToString("D6", System.Globalization.CultureInfo.InvariantCulture);
     }
     ```

3. CƠ CHẾ XÁC THỰC & ỦY QUYỀN VỚI JWT & REFRESH TOKEN
   - File tham chiếu: JwtTokenService.cs (Tầng Infrastructure)
   - Nguyên lý:
     + Access Token (JWT): Token ngắn hạn (thường sống 15 - 60 phút) chứa thông tin định danh và phân quyền của User dưới dạng các Claim (userId, email, role). Token này được gửi kèm theo mỗi request của client trong HTTP Header (Authorization: Bearer <Token>). Hệ thống chỉ cần giải mã kiểm tra chữ ký số HMAC-SHA256 của token bằng khóa bí mật (Secret Key) để cho phép truy cập mà không cần truy vấn Database nhiều lần.
     + Refresh Token: Được lưu trong Database, có thời hạn sống dài hơn (ví dụ: 7 đến 30 ngày). Khi Access Token hết hạn, Client gửi Refresh Token lên để xin cấp mới Access Token mà không bắt người dùng phải nhập lại mật khẩu.
     + Kỹ thuật Token Rotation (Xoay vòng Token): Mỗi lần sử dụng Refresh Token để lấy Access Token mới, hệ thống sẽ hủy Refresh Token cũ và cấp thêm một Refresh Token mới. Nếu kẻ trộm đánh cắp được Refresh Token cũ và sử dụng lại, hệ thống sẽ phát hiện hành vi tái sử dụng (Reuse) và lập tức thu hồi toàn bộ các phiên đăng nhập của User đó để bảo vệ tài khoản.

   - Đoạn Code tạo Refresh Token ngẫu nhiên (64 bytes an toàn tuyệt đối):
     ```csharp
     public string GenerateRefreshToken()
     {
         return Convert.ToBase64String(RandomNumberGenerator.GetBytes(64));
     }
     ```

4. CƠ CHẾ XỬ LÝ LỖI TẬP TRUNG (GLOBAL EXCEPTION MIDDLEWARE)
   - File tham chiếu: GlobalExceptionMiddleware.cs (Tầng API/Middlewares)
   - Nguyên lý:
     Thay vì sử dụng các khối `try-catch` lặp đi lặp lại ở mọi Controller, dự án thiết lập một Middleware chặn ở cổng vào HTTP Pipeline của ASP.NET Core để bắt tất cả các ngoại lệ (Exception) chưa được xử lý. Điều này đảm bảo:
     + Backend không bao giờ bị sập (Crash) hoàn toàn.
     + Không lộ dấu vết ngăn xếp code (Stack trace) chi tiết ra ngoài màn hình client (nguy cơ bảo mật lớn).
     + Định dạng phản hồi lỗi luôn đồng nhất với cấu trúc chung: `{ success, message, data, errorCode, errors }`.
     + Ánh xạ chính xác các Exception nghiệp vụ thành HTTP Status Code phù hợp (400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Internal Error).

   - Đoạn Code quan trọng:
     ```csharp
     private static Task WriteErrorResponseAsync(HttpContext context, Exception exception)
     {
         var (statusCode, message, errorCode) = exception switch
         {
             UnauthorizedAccessException => (HttpStatusCode.Unauthorized, "Unauthorized.", "UNAUTHORIZED"),
             ArgumentException => (HttpStatusCode.BadRequest, exception.Message, "BAD_REQUEST"),
             InvalidOperationException => (HttpStatusCode.BadRequest, exception.Message, "BAD_REQUEST"),
             KeyNotFoundException => (HttpStatusCode.NotFound, exception.Message, "NOT_FOUND"),
             _ => (HttpStatusCode.InternalServerError, "An unexpected error occurred.", "INTERNAL_SERVER_ERROR")
         };
         
         context.Response.ContentType = "application/json";
         context.Response.StatusCode = (int)statusCode;
         var response = ApiResponse<object>.Fail(message, errorCode);
         return context.Response.WriteAsync(JsonSerializer.Serialize(response, ...));
     }
     ```

TẬP 4: THUỘC TÍNH (ATTRIBUTES) & RÀNG BUỘC DỮ LIỆU TRONG DTO
================================================================================
Để bảo vệ hệ thống trước dữ liệu rác hoặc mã độc gửi từ Client, các lớp DTO trong dự án sử dụng các thuộc tính (Attributes/Annotations) kiểm tra dữ liệu ở mức khai báo:

Ví dụ DTO đăng ký tài khoản (RegisterRequest):
```csharp
public class RegisterRequest
{
    [Required(ErrorMessage = "Email là bắt buộc.")]
    [EmailAddress(ErrorMessage = "Định dạng email không hợp lệ.")]
    [StringLength(100, ErrorMessage = "Email không được vượt quá 100 ký tự.")]
    public string Email { get; set; } = null!;

    [Required(ErrorMessage = "Mật khẩu là bắt buộc.")]
    [StringLength(100, MinimumLength = 8, ErrorMessage = "Mật khẩu phải từ 8 đến 100 ký tự.")]
    [RegularExpression(@"^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[^\da-zA-Z]).{8,}$", 
        ErrorMessage = "Mật khẩu phải chứa ít nhất 1 chữ hoa, 1 chữ thường, 1 số và 1 ký tự đặc biệt.")]
    public string Password { get; set; } = null!;
}
```

* Giải thích các thuộc tính:
- `[Required]`: Bắt buộc thuộc tính không được null hoặc để trống.
- `[EmailAddress]`: Tự động kiểm tra chuỗi có tuân thủ định dạng email chuẩn hay không (chứa dấu @ và tên miền hợp lệ).
- `[StringLength]`: Giới hạn độ dài tối đa/tối thiểu của chuỗi nhằm ngăn ngừa lỗi tràn bộ đệm (Buffer overflow) hoặc hacker gửi chuỗi siêu lớn làm treo bộ nhớ server.
- `[RegularExpression]`: Áp dụng biểu thức chính quy (Regex) để bắt buộc mật khẩu phải mạnh, tăng tính bảo mật tài khoản.
